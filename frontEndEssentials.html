Script in Head vs Body

(nothing placed in the Head should be visible to the user)

When the browser encounters the script tag, it will block progress on parsing the rest of the HTML on the page as it downloads (if external) the script file and executes it.

In the past, this meant putting script tags at the bottom of the body.

The modern approach is to use async or defer.

While the browser is downloading the script, it is also continuing to parse the HTML. It will execute the script as soon as it is done downloading.

Keep in mind, async scripts are not guarenteed to run in a specific order, and you should not use document.write. Instead, add event listeners:

defer:

The browser will download and execute the scripts in order after the document has been loaded.

What makes a website slow?

  • Place CSS tags before JavaScript tags: Load CSS before JS because the browser will stop to load the script before CSS can be displayed
  • Minify HTML, CSS, and JS: Remove whitespace and new lines to reduce size and speed up load time. Significant for CSS/JS, as this means less data is sent to the client
  • Critical CSS: In-line code the CSS needed to load the first display that the user sees, then load non-critical asynchronously (the JS function is a third-party that loads CSS aynchronously):
  • Unoptimized Images: Use CSS or SVG instead if possible. JPEG is usually faster (more compressed).
  • HTTPS is slower because it's more secure, so browsers allow more options (geolocation, push notifications, etc.)
  • Make fewer requests by combining external CSS, JS, etc. files
  • Cache-Control: A HTTP header that allows you to cache things (like images, CSS/JS files, etc)
  • Prefetching: Tell browser to load other pages in advance after the main one is done, in case the user navigates to it next. (rel="prefetch")
  • Use a CDN: Content Delivery Network, edge servers all over the world so a user can fetch content from the one closest to them. Edge servers just store a cache of the pages

Web Architecture

  • Basically, your browser asks your router which asks your ISP (Comcast, etc) which searches the DNS (Domain Name System), which finds the computer the desired website is on and returns it to you to connect to. The DNS is like a phonebook: looks up the requested domain name and returns the IP address of the machine it's on.
  • Request - an ASCII-encoded text sent to a server:
    POST https://postman-echo.com/get?arg1=value&arg2=query2#fragment_id
    • Verb/Request Method: GET, POST, PUT, DELETE. (also HEAD, etc.)
      • Safe vs Unsafe: GET and HEAD are safe (do not change anything), while others are unsafe
      • Idempotent vs Not Idempotent: GET, HEAD, PUT, and DELETE are idempotent, meaning making the same request multiple times will have the same end result
    • Scheme/Protocol: tells the browser how to send the request, and how to make sense of the response
      • http:// - HyperText Transfer Protocol
      • https:// - HyperText Transfer Protocol Secure
      • ws:// - websocket
      • ftp:// - file transfer protocol
    • Hostname: assets.company.com
      • subdomain: assets.company.com
      • domain: assets.company.com (this is the only required one)
      • top level domain (tld): assets.company.com
    • Port: .com(:443)/ ...
      • Every request has a port, even if hidden
      • They allow things on the internet to talk to things on the server
      • http assumes port 80
      • https assumes port 443
      • When developing on local server (http://localhost:8080), try to use a non-privileged port. Ports 1-1023 are special (privileged)
    • Path: anything after the url and before the ? is the path. It says where to go.
    • Query Arguments/Strings: ?arg1=value&arg2=query2
      • Start with ?, key/value pairs of arg=value, each separated with &
      • If there's an = without a value, the key will exist and the value will be an empty string
      • Used to filter the result, for example a site filtering 4k monitors by Dell: .com/monitors?res=4k&make=Dell
    • Fragment Identifier: #fragment_id like an anchor on a page. The server isn't told this information. Some apps use this to display parts of the app without needing to make network requests.
    • Headers: not visible in a URL, but it contains information on cookies, the host (domain), etc.
    • Body: only for POST, PUT, PATCH, (and some) DELETE requests, the body is the "what", or the content of the request.
  • Response
    • First line is the protocol version, status code, and status line
    • Next is the response headers.
    • Then a blank line separating metadata (top) and the body content (bottom).
    • Body can be a JSON payload if sending the request to an API
    • Responses can be from either the web server or the application
    • Status Code: what the server did with your request
      • 200 OK, 201 Created, 204 No Content - Generally are good, means everything went as expected
      • 301 Moved Permanently, 302 Found, 304 Not Modified (when you're downloading something but already have a local copy of it cached) - Redirection. Things weren't where you thought it was
      • 400 Bad Request (request is not understood), 401 Unauthorized (need to log in), 403 Forbidden (logged in, no permission), 404 Not Found (asking for something that's not there), 405 Method Not Allowed (e.g. sending POST when the endpoint only accepts GET) - server is ok, but you need to modify the request
      • 500 Internal Server Error (code error), 502 Bad Gateway (code error), 503 Service Unavailable (server is too busy), 504 Gateway Timeout (server is too slow) - Server is failing
  • SSL (now TLS) - Secure Socket Layer (or Transport Layer Security)
    • TLS is the modern version of SSL, but since the terminology got stuck, SSL is now referred to for both SSL + TLS
    • HTTPS = HTTP + TLS:
      HTTPS is an implementation of TLS on HTTP to secure the connection
    • TLS is needed because on old HTTP sites that haven't been upgraded yet, the connection is insecure, leading hackers to get your data if they have access to your router info, or your ISP can inject ads, etc.
    • TLS provides integrity (content cannot be tampered with), encryption (content cannot be read by others), and authentication (you connect to the server you should be on)
    • The Process:
      1. The browser requests encryption from the server, and sends the encryption protocols it supports
      2. Server agrees, and selects a protocol they can both use. It also sends their SSL certificate (to prove that they're them), and a random string it just generated
      3. Browser double-checks the certificate, and sends a generated random string
      4. Both ends have both strings. The client has the server's PUBLIC key it got from the certificate, and the server has their own PRIVATE key. Using these, both sides should arrive at the same generated code
      5. Browser tells server what code it got, and vice versa
      6. If they match, that code is now the key for the session, and everything else will be encrypted with that key
      7. Whenever the client asks for something, the request will be encrypted with that key and unintelligible to anyone else that may be looking
      8. Similarly, the browser's responses will be encrypted
  • Domain - human-friendly address of a website. Computers communicate sites using IP addresses, while people use domains
  • Registrar
    • Registry - a list that keeps track of Top Level Domains (.com, .co, .edu, etc)
